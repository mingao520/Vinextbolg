春节期间，我花了四个小时，从零开始 Vibe Coding 了一个网络流量分析面板 [Neko Master](https://github.com/foru17/neko-master)，四个小时上线第一版，项目上线一周，GitHub 收获了 1000+ Star，Docker Pull 破了 10K。

如果你家里也在用 OpenClash / Mihomo / Surge，欢迎体验：

- **GitHub**: [github.com/foru17/neko-master](https://github.com/foru17/neko-master)
- **在线演示**: [neko-master.is26.com](https://neko-master.is26.com)（`密码: neko2026`）

MIT 开源，欢迎 Star 和提 Issue。


从最初的「玩具」到现在拥有复杂架构的项目，这篇文章，算是对整个开发过程的一个回顾和总结，分享一些 Vibe Coding 的实战体感。

## 一.为什么会有这个项目

我是一个 Homelab 用户，家里跑了一堆服务，分流策略比较复杂，日常开发也会频繁切换 IP。

其实早在 2024 年初，我就折腾过一次流量监控方案：用 Grafana 和 Loki 配合 Clash Premium 的 Tracing API，弄了一个监控面板。当时发了条推，说「能看看自己的线路流量什么的，其实也没啥用」。

<TweetCard tweetId="1744017015256346724" />

但用了一段时间后，发现 Grafana 这套东西虽然功能强大，对于家庭用户来说门槛还是太高了。配置繁琐，界面也不够直观，更关键的是——长得不好看。原生的 Clash 面板更多是「运行状态」展示，但我一直缺少一个更直观的视角去看：

**流量到底在干什么？**

- 哪些域名在吃带宽？
- 哪个节点负载更高？
- 深夜学习 AI 时到底哪个网站最耗流量？
- 当前的规则策略是否合理？

市面上除了 UniFi 之外，其他统计工具的界面确实有些一言难尽。与其在不同工具之间拼凑数据，不如自己做一个更好看、更好用的「流量感知」的面板。

## 二.一小时 MVP

2 月 5 日下午，我打开 [Kimi Code](https://www.kimi.com/code)，使用最新的 K2.5 模型，开始 Vibe。

没有画原型图，没有写技术方案，就是在脑子里想了个大概，然后开始跟 AI 对话。一小时不到，MVP 就跑起来了——部署在内网，监听 OpenClash 流量，能看到域名统计、节点流量、IP 归属地查询，数据还能持久化。

当天晚上又花了三个小时打磨了一下，凌晨四点半，第一版 Clash Master 上线。

不到 24 小时，项目收获了 GitHub 400+ Star，Docker Hub 1000+ 次拉取。这就是 AI 时代的开发速度。

## 三.去 AI 味

很多人吐槽 AI 生成的界面「千篇一律」，说实话第一版确实如此。分享一个我实践下来觉得很有效的技巧：

> **不要只用文字描述「给我写个好看的面板」。要给视觉锚点。**

具体做法：去 Dribbble / Figma Community 搜「Dashboard」，挑一张看着舒服的截图直接喂给 AI，告诉它「复刻这个设计感」。配色、卡片阴影、数据可视化的风格，都可以用截图来锚定。

有了参考系，AI 的审美直接从「程序员风」进化到「SaaS 级」。

<TweetCard tweetId="2019758733983912111" />

一个我越来越确信的感受是：**在 AI 时代，代码的门槛在下降，但审美判断依然是决定成品质量的关键因素。** AI 可以帮你写代码、做布局、调样式，但「好不好看」这件事，最终还是得靠人来判断。

## 四.AI 工具复盘

这个项目开发过程中，我把市面上主流的几个 AI 编码工具都用了个遍。直接说结论：

| 工具 | 角色 | 体感 |
|------|------|------|
| **Kimi K2.5** | 早期主力 | 量大管饱，中文理解好，MVP 阶段 100% 主力，甚至没见过任何限额提示 |
| **Claude Code (Opus 4.6)** | 硬骨头 | 贵公子。一个复杂任务下去 4 小时限额 15 分钟直接见底，但遇到架构和性能深水区，只有它能啃 |
| **CodeX (GPT 5.2/5.3)** | 日常输出 | 5 小时循环用量非常扎实，开发过程基本碰不到小时限制的瓶颈，但周限量两天就用完了 |
| **Gemini 3 Pro** | 辅助 | 主要用来 Review 和写 Commit Message，质量偶尔掉线 |

一个真实体感：**国产模型在初始化阶段已经足够高效**，Kimi K2.5 是 Clash Master 诞生阶段的绝对主力。但当项目复杂度提升，面对数据库性能调优、架构重构这些深水区问题时，还是得靠 Claude Opus 4.6 和 CodeX 5.3 交叉 Debug。

开发到第五天，四家 AI 的周额度全部被我榨干了。

<TweetCard tweetId="2020143157279617155" />

没有 AI 可用的程序员，大概和没有咖啡的早晨差不多。

但这也逼着我思考了一个问题：**当你的 Vibe Coding 工作流依赖多个 AI 服务时，「额度管理」本身也是一种工程能力。** 哪些任务值得用贵的模型，哪些用量大管饱的就够了，这个判断需要你对任务复杂度有感知。不然要么额度见底项目停摆，要么用错了模型效率打折。

## 五.Vibe Coding 的节奏：快速迭代与用户反馈

上线之后的两周，基本就是一个循环：**发版 → 收反馈 → 修 Bug → 加功能 → 发版。** 从 v1.0.2 到 v1.3.2，迭代了大约 20 个版本。

这个节奏下，AI 的角色不再是「帮我从零写一个功能」，而是变成了「帮我快速响应用户反馈」。V2EX 上有人说磁盘 I/O 炸了，我把日志贴给 Claude，十五分钟定位到是 SQLite 单条写入没做批处理，连夜修了三个版本，I/O 从一天 200GB 降到了 1.6GB。Docker 镜像太大被吐槽，让 AI 帮我做多阶段构建，从 800MB 瘦到 300MB。

<TweetCard tweetId="2019675670914428975" />

这个阶段的一个核心体感是：**Vibe Coding 不只是「让 AI 写代码」，更是「让 AI 帮你加速整个反馈循环」。** 用户提了个需求，你不需要花半天去查文档、写方案，直接跟 AI 说清楚上下文，几分钟就能拿到一个可用的 patch。这种响应速度，在开源项目的早期阶段是非常关键的——用户看到你迭代快，信任感就上来了。

## 六.从 Clash Master 到 Neko Master

项目最初叫 Clash Master，用了一周后改名为 **Neko Master**。原因很简单：不想跟 Clash 这个名字绑定太死，后来支持了 Surge v5+，未来也可能接入更多网关类型。

<TweetCard tweetId="2020829243991806432" />

改名的同时做了一次比较大的架构升级，包括 Agent 分布式部署模式、ClickHouse 大规模分析后端、本地 MMDB 离线 IP 查询等。如果你对技术细节感兴趣，GitHub 仓库里有完整的[架构文档和部署说明](https://github.com/foru17/neko-master)，这里就不展开了。

<TweetCard tweetId="2021816174447919603" />

<TweetCard tweetId="2022944879853342987" />

简单说两个我觉得比较有意思的设计决策：

- **Agent 模式**：让数据采集器部署在网关旁边主动上报，面板不需要直连网关。这个思路不是 AI 想出来的，是我自己家里多网关、跨网段的真实痛点驱动的。
- **WebSocket + HTTP 回退**：打得通 WebSocket 就实时推送，打不通就自动降级到轮询，用户无感。这种「优雅降级」的方案，也是在社区反馈中打磨出来的。

## 七.一些数据和观察

项目上线两周的一些数据：

- **GitHub Star**: 1400+
- **Docker Pull**: 10K+
- **版本迭代**: v1.0.2 → v1.3.2（约 20 个版本）
- **主要 Issue**: 大部分是配置和部署问题

一个有意思的观察：**大部分用户其实不是开发者**，而是普通的 Homelab 爱好者。他们更关心「能不能一键跑起来」「占多少资源」「会不会影响主路由」，远多于「用了什么技术栈」「支不支持自定义主题」。

这也让我调整了项目的优先级：**易用性 > 功能丰富度**。与其加更多花哨的功能，不如把部署流程和默认配置打磨好。Docker 一把梭，暴露一个 3000 端口就能跑起来，这是我希望达到的体验。

## 八.写在最后

从 2024 年用 Grafana 弄的那个「其实也没啥用」的面板，到现在 Neko Master 被一千多人用着，中间隔了两年。但真正写代码的时间，加起来可能也就两周。

Vibe Coding 确实改变了我的开发方式。回顾这两周，我的工作流大致是这样的：

1. **想清楚要什么** — 不画原型，但脑子里要有画面
2. **找视觉锚点** — Dribbble 截图比一千字 prompt 管用
3. **选对模型干对活** — 初始化用 Kimi，深水区用 Claude，日常迭代用 CodeX
4. **快速发版收反馈** — 用户的真实场景比你的想象更重要
5. **人做判断，AI 做执行** — 审美、架构决策、优先级排序，这些是人的活

这套流程跑下来，过去需要一两周的原型，现在几小时就能跑起来。但有一个东西 AI 替代不了：**从「能跑」到「好用」的那段距离。**

200GB 的写入 bug 是 AI 写的，但发现问题、定位瓶颈、设计缓存策略是人做的。界面从「AI 味」到「SaaS 级」，靠的不是更好的 prompt，而是你自己对美的判断。Agent 模式的架构设计，来自对真实部署场景的理解，不是 AI 能凭空想出来的。

Vibe Coding 降低了「从 0 到 1」的门槛，但「从 1 到 100」的路，依然需要经验、审美和对用户需求的理解。
