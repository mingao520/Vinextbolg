上个月，我花了两周时间，从零开始 Vibe Coding 了一个网络流量分析面板 [Neko Master](https://github.com/foru17/neko-master)。项目上线一周，GitHub 收获了 1400+ Star，Docker Pull 破了 10K。

这篇文章，算是对整个开发过程的一个回顾和总结，聊聊从「脑子里有个想法」到「一千多人在用」的经历，也顺便分享一些 Vibe Coding 的实战体感。

![](https://c2.is26.com/blog/2026/03/neko-master/demo-1.jpg)

## 一.为什么会有这个项目

我是一个 Homelab 用户，家里跑了一堆服务，分流策略比较复杂，日常开发也会频繁切换 IP。

其实早在 2024 年初，我就折腾过一次流量监控方案：用 Grafana 和 Loki 配合 Clash Premium 的 Tracing API，弄了一个监控面板。当时发了条推，说「能看看自己的线路流量什么的，其实也没啥用」。

![](https://c2.is26.com/blog/2026/03/neko-master/grafana-old.jpg)

但用了一段时间后，发现 Grafana 这套东西虽然功能强大，对于家庭用户来说门槛还是太高了。配置繁琐，界面也不够直观，更关键的是——长得不好看。原生的 Clash 面板更多是「运行状态」展示，但我一直缺少一个更直观的视角去看：

**流量到底在干什么？**

- 哪些域名在吃带宽？
- 哪个节点负载更高？
- 深夜学习 AI 时到底哪个网站最耗流量？
- 当前的规则策略是否合理？

市面上除了 UniFi 之外，其他统计工具的界面确实有些一言难尽。与其在不同工具之间拼凑数据，不如自己做一个更聚焦「流量感知」的面板。

## 二.一小时 MVP

2 月 5 日下午，我打开 Kimi K2.5，开始 Vibe。

没有画原型图，没有写技术方案，就是在脑子里想了个大概，然后开始跟 AI 对话。一小时���，MVP 就跑起来了——部署在内网，监听 OpenClash 流量，能看到域名统计、节点流量、IP 归属地查询，数据还能持久化。

当天晚上又花了三个小时打磨了一下，凌晨四点半，第一版 Clash Master 上线。

![](https://c2.is26.com/blog/2026/03/neko-master/v1-vs-v2.jpg)

这里有一个有意思的对比：图一是当晚 Vibe 出来的第一版，图二是几天后的完全体。前者 AI 味浓浓，后者审美基本达到了 2026 年现代 App 的水准。

不到 24 小时，项目收获了 GitHub 400+ Star，Docker Hub 1000+ 次拉取。这就是 AI 时代的开发速度。

## 三.去 AI 味

很多人吐槽 AI 生成的界面「千篇一律」，说实话第一版确实如此。分享一个我实践下来觉得很有效的技巧：

> **不要只用文字描述「给我写个好看的面板」。要给视觉锚点。**

具体做法：去 Dribbble / Figma Community 搜「Dashboard」，挑一张看着舒服的截图直接喂给 AI，告诉它「复刻这个设计感」。配色、卡片阴影、数据可视化的风格，都可以用截图来锚定。

有了参考系，AI 的审美直接从「程序员风」进化到「SaaS 级」。

![](https://c2.is26.com/blog/2026/03/neko-master/design-evolution.jpg)

一个我越来越确信的感受是：**在 AI 时代，代码的门槛在下降，但审美判断依然是决定成品质量的关键因素。** AI 可以帮你写代码、做布局、调样式，但「好不好看」这件事，最终还是得靠人来判断。

## 四.AI 工具复盘

这个项目开发过程中，我把市面上主流的几个 AI 编码工具都用了个遍。直接说结论：

| 工具 | 角色 | 体感 |
|------|------|------|
| **Kimi K2.5** | 早期主力 | 量大管饱，中文理解好，MVP 阶段 100% 主力，甚至没见过任何限额提示 |
| **Claude Code (Opus 4.6)** | 硬骨头 | 贵公子。一个复杂任务下去 4 小时限额 15 分钟直接见底，但遇到架构和性能深水区，只有它能啃 |
| **CodeX (GPT 5.2/5.3)** | 日常输出 | 5 小时循环用量非常扎实，开发过程基本碰不到小时限制的瓶颈，但周限量两天就用完了 |
| **Gemini 3 Pro** | 辅助 | 主要用来 Review 和写 Commit Message，质量偶尔掉线 |

一个真实体感：**国产模型在初始化阶段已经足够高效**，Kimi K2.5 是 Clash Master 诞生阶段的绝对主力。但当项目复杂度提升，面对数据库性能调优、架构重构这些深水区问题时，还是得靠 Claude Opus 4.6 和 CodeX 5.3 交叉 Debug。

开发到第五天，四家 AI 的周额度全部被我榨干了。当时发了条推：「这周剩下的时间可咋办🥹」——没有 AI 可用的程序员，大概和没有咖啡的早晨差不多。

## 五.踩坑纪实

项目虽小，但踩的坑不少。挑几个印象深刻的说说。

### 5.1 一天 200GB 的数据库写入

第一版的 SQLite 写入逻辑有问题。用户部署上去之后，磁盘 I/O 爆炸，一天写入量高达 **200GB**。V2EX 上有哥们反馈「硬盘灯从没灭过」，给我吓了一跳。

紧急排查后发现是每条流量日志都在做单条 INSERT，没有做任何批处理和缓存。连续迭代了几个版本：

```
v1.0.2 → v1.0.7 → v1.0.9
  ↓       ↓        ↓
200GB    20GB    1.6GB/天
```

引入内存缓存 + 30 秒批量 flush + 写入限流后，I/O 负载降低了 **99%**。这个坑说白了就是基本功问题，但在 Vibe Coding 的节奏下，AI 生成的代码很容易忽略这类性能细节——它能帮你把功能写出来，但不一定会帮你考虑「这东西跑在用户的 N5105 NAS 上会怎样」。

### 5.2 Docker 镜像瘦身

早期镜像 **800MB+**，被用户吐槽不少。后来用 `pnpm deploy` 做依赖精简 + 多阶段构建，镜像降到了 **300MB+**。对于一个 Node.js + Next.js 的项目来说，这个体积还算合理。

### 5.3 WebSocket 与 HTTP 回退

早期版本只有 HTTP 轮询（5 秒一次），数据刷新延迟明显。后来加了 WebSocket 支持，实时性大幅提升。

但 WebSocket 在反向代理层需要额外配置（Nginx 的 `proxy_set_header Upgrade`），对于很多非技术用户来说是个门槛。最终的方案是保留了 HTTP 轮询作为 fallback——打不通 WebSocket 就自动降级，用户无感。

## 六.从 Clash Master 到 Neko Master

项目最初叫 Clash Master。用了一周后改名为 **Neko Master**。

改名的原因主要是不想跟 Clash 这个名字绑定太死——项目后来支持了 Surge v5+，未来也可能支持更多网关类型。Neko（猫）这个名字，来自日语，寓意是「安静但敏锐」，也算是对这个工具定位的一种表达：轻量、精准、不喧宾夺主。

改名的同时也做了一次比较大的架构升级：

### Agent 模式

这是 v1.3.2 引入的一个重大更新。

之前的方案是面板直连网关——这意味着你的 Neko Master 部署机器需要能访问到 Clash/Surge 的 API 端口。在简单的家庭网络里这没问题，但如果你有多个网关、或者网关和面板不在同一个网段，就会比较头疼。

Agent 模式引入了一个反向的思路：

- Agent 部署在网关旁边（OpenWrt、Linux、macOS），主动拉取数据并上报到面板
- 面板不需要主动连接网关，网络隔离更清晰
- 支持多实例，同一台机器可以跑多个 Agent 监控不同网关

配套还写了个 `nekoagent` CLI 管理器，支持 `add / start / restart / logs / update / remove` 全生命周期管理。做运维的同学应该会喜欢这个。

### ClickHouse 分析后端

SQLite 对于单网关家庭用户完全够用。但如果是多 Agent、多网关的场景，域名和 IP 的数据量可以到十万甚至百万级别，SQLite 的查询性能就开始吃力了。

v1.3.2 新增了 ClickHouse 作为可选的分析后端，面向大规模场景做高性能写入和查询。支持纯 ClickHouse 模式，也支持双写模式（SQLite + ClickHouse），方便渐进迁移。

## 七.部署建议

在 V2EX 和 Issue 里回答最多的问题就是部署相关的，这里统一说一下。

**强烈建议旁路部署。** 不要和网关（OpenWrt / 软路由）部署在同一台机器上。

虽然优化后 I/O 已经大幅降低，但这个项目有持续的数据读写，万一出问题影响主路由，家里网络就仆街了。

我自己是部署在 NAS（N5105 CPU）上，监控 3 个上游 Clash 网关，目前运行稳定，对主网关的负载影响也比较小。

最简部署，Docker 一把梭：

```bash
docker pull foru17/neko-master
```

只暴露 3000 端口就能跑起来。如果想要 WebSocket 实时推送，需要反代层把 3002 端口也打通；打不通就回退到 HTTP 轮询（约 5 秒级），也不影响使用。

## 八.一些数据和观察

项目上线两周的一些数据：

- **GitHub Star**: 1400+
- **Docker Pull**: 10K+
- **版本迭代**: v1.0.2 → v1.3.2（约 20 个版本）
- **主要 Issue**: 大部分是配置和部署问题

一个有意思的观察：**大部分用户其实不是开发者**，而是普通的 Homelab 爱好者。他们更关心「能不能一键跑起来」「占多少资源」「会不会影响主路由」，远多于「用了什么技术栈」「支不支持自定义主题」。

这也让我调整了项目的优先级：**易用性 > 功能丰富度**。与其加更多花哨的功能，不如把部署流程和默认配置打磨好。

## 九.写在最后

从 2024 年用 Grafana 弄的那个「其实也没啥用」的面板，到现在 Neko Master 被一千多人用着，中间隔了两年。但真正写代码的时间，加起来可能也就两周。

AI 确实改变了开发方式。过去需要一两周的原型，现在几小时就能跑起来。Kimi 一小时出 MVP，Claude 啃深水区的性能问题，CodeX 做日常迭代——多模型协作，已经是我现在的常态工作流了。

但有一个东西 AI 替代不了：**从「能跑」到「好用」的那段距离。**

200GB 的写入 bug 是 AI 写的，但发现问题、定位瓶颈、设计缓存策略是人做的。界面从「AI 味」到「SaaS 级」，靠的不是更好的 prompt，而是你自己对美的判断。Agent 模式的架构设计，来自对真实部署场景的理解，不是 AI 能凭空想出来的。

Vibe Coding 降低了「从 0 到 1」的门槛，但「从 1 到 100」的路，依然需要经验、审美和对用户需求的理解。
